# Governance

`frame-by-frame` uses a maintainer-led governance model while the project and its community are small.

## Roles

### Contributors

Anyone who reports an actionable problem, improves documentation, participates constructively in design discussions, or submits code is a contributor.

### Maintainers

Maintainers review contributions, protect the project's scope and quality, moderate community spaces, manage releases, and make final decisions when consensus is not possible. The repository owner is the initial maintainer.

Additional maintainers may be invited based on sustained, constructive contributions; sound technical judgment; reliable reviews; and consistent adherence to the Code of Conduct.

## Decision process

Routine changes are decided through issue and pull request review. Significant changes to public APIs, compatibility, architecture, accessibility defaults, governance, or release policy should begin with a public issue and include the problem, alternatives, and trade-offs.

Maintainers seek relevant feedback and prefer rough consensus. When a decision is required, maintainers have final authority and should publish the rationale. Security reports, conduct cases, and other sensitive matters remain private where disclosure would cause harm.

## Releases

Maintainers decide when a release meets its documented acceptance criteria. Release notes should describe user-visible changes, breaking changes, known limitations, and migration steps where applicable.

## Changes to governance

This model may evolve as the contributor base grows. Material governance changes will be proposed publicly and recorded in this document.
